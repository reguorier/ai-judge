# 跨界架构探讨：天府Agent 设计模式 → AI Judge v3.1.0 的可迁移性

> **议题目标**：以天府Agent（商业闭源 AI 推理产品）的架构设计为镜，探讨其对 ai-judge-v3.1.0-harness（本地桌面评判框架）的可借鉴价值，形成一份候选升级方案清单，供 AI Judge 团队评估其现有能力覆盖度与下一步方向。

---

## 一、天府Agent 是什么

天府Agent（Tianfu Agent）是 DestinyLinker 公司开发的商业 AI 产品，定位为「全球首个命盘推演智能体」。它融合紫微斗数、子平八字、奇门遁甲三大传统术数体系，核心卖点不是「算得准」，而是**让用户看到推理全貌——从输入到结论的每一步都可追溯、可质疑、可验证**。

### 1.1 架构总览

天府Agent 是一个五层系统：

```
┌──────────────────────────────────────┐
│          UI 层：可视化推理链            │
│  命盘图 / 推理步骤 / 置信度热力 / 溯源  │
└────────────────┬─────────────────────┘
                 │
┌────────────────▼─────────────────────┐
│     编排层：Multi-Agent 协作网络        │
│  排盘Agent │ 推理Agent×3 │ 决策Agent │ 报告Agent │
└────────────────┬─────────────────────┘
                 │
┌────────────────▼─────────────────────┐
│      推理层：思考-验证-反馈 闭环         │
│  Think → Verify → Feedback (RLHF)    │
└──────┬──────────────────┬────────────┘
       │                  │
┌──────▼──────┐  ┌────────▼──────────┐
│ 私有化命理   │  │ 自研嵌入模型       │
│ 垂直LLM     │  │ (天干地支/五行编码) │
└──────┬──────┘  └────────┬──────────┘
       │                  │
┌──────▼──────────────────▼──────────┐
│     知识库层：纯净古籍 + 长期记忆     │
│  规则溯源 / 时间线锚定 / 用户档案     │
└────────────────────────────────────┘
```

### 1.2 关键设计模式（可迁移部分）

| 设计模式 | 说明 | 对应的通用问题 |
|---------|------|--------------|
| **多智能体合议** | 4 类 Agent 分工协作而非单一模型输出 | 复杂判断需要多视角交叉验证 |
| **思考-验证-反馈闭环** | Think→Verify→Feedback，内置异议机制 | AI 判断需要自检和持续进化 |
| **GPRO 概率推理** | 对每个推理节点输出置信度，拒绝绝对化 | 不确定性的诚实表达比硬判断更可信 |
| **知识溯源** | 每个结论标注古籍出处或方法论依据 | 用户信任建立在「有据可查」之上 |
| **可视化推理链** | 从输入到结论的完整步骤在 UI 上展开 | 黑盒输出 → 可审计的推理过程 |
| **250+ 领域工具** | 排盘、历法、五行等专用 function calling | 通用模型 + 领域工具矩阵 = 专业输出 |
| **长期记忆锚定** | 用户档案 + 时间线 → 个性化推演 | 上下文积累使判断越来越贴合用户 |

### 1.3 该产品不开源

天府Agent 是闭源商业产品。同公司唯一公开的 GitHub 仓库是 `DestinyLinker/MingLi-Bench`（MIT 协议），为一个 160 道选择题的命理推理评测框架，含 CLI、多 Provider 路由、并发评测引擎等基础设施代码，可作为该团队工程风格的参考样本。

## 二、本地 AI Judge（ai-judge-v3.1.0-harness）的定位假设

由于未直接阅读 `ai-judge-v3.1.0-harness.tar.gz` 的源码，以下为基于文件名和上下文的结构化推测，**请在评估时以实际代码为准，修正或推翻这些假设**。

### 2.1 推测的组件构成

```
ai-judge-v3.1.0-harness/
├── harness/          # 测试/评判执行框架
│   ├── runner        # 执行器：调度评判任务
│   ├── collector     # 收集器：汇总评判结果
│   └── reporter      # 报告器：输出评判报告
├── rules/            # 规则定义
├── analyzers/        # 各类分析器（lint/SAST/test/自定义）
├── models/           # 模型调用层
└── ui/               # 桌面端界面
```

### 2.2 推测的核心流程

```
输入（代码PR / 文档 / 配置变更）
    │
    ▼
规则匹配 + 分析器执行
    │
    ▼
模型推理（评分 / 分类 / 建议生成）
    │
    ▼
结果汇总 → 报告输出
```

### 2.3 推测的当前局限（作为讨论靶子）

以下局限是**刻意提出的假设性批评**，目的是激发讨论，不代表实际缺陷：

- **假设 1**：评判是单一模型的一次性输出，缺乏多视角合议
- **假设 2**：输出以评分/标签为主，缺少完整推理链和证据溯源
- **假设 3**：缺乏异议/对抗验证机制，可能漏判或误判
- **假设 4**：用户反馈没有形成闭环，模型不会因用户纠错而进化
- **假设 5**：确定性判断为主，缺少不确定性量化和置信度分级
- **假设 6**：桌面端 UI 没有充分利用本地交互优势（推理链可点击展开、直接跳转源文件等）

## 三、需要 AI Judge 团队探讨的方向

以下每个方向都包含：**天府Agent 的做法 → 对 AI Judge 的可能映射 → 需要团队回答的关键问题**。

---

### 方向 A：多智能体合议 vs 单一评判引擎

**天府的做法**：4 类 Agent（排盘、推理×3、决策、报告）分工协作，非单一模型。

**对 AI Judge 的映射**：

```
当前（假设）                    候选方案
┌──────────┐          ┌────────────────────────┐
│ 单一评判   │    →    │ 合议庭编排器             │
│ 模型      │          │ ├ 事实认定Agent          │
│          │          │ ├ 规则匹配Agent          │
└──────────┘          │ ├ 判例检索Agent          │
                      │ ├ 异议Agent（反方）       │
                      │ └ 报告生成Agent          │
                      └────────────────────────┘
```

**需要团队回答**：
1. 当前 AI Judge 的评判是单模型单次推理，还是已有内部的多步分解？
2. 多 Agent 合议会显著增加推理延迟和成本——在本地桌面的资源约束下，这个 tradeoff 可接受吗？是否可以按任务严重性做分级（critical 任务走合议，trivial 任务走快判）？
3. AI Judge 是否已有「异议/反方」机制？比如对同一个判断自动生成反方论点？
4. 判例检索：是否有历史评判结果的向量化存储和相似案例召回能力？

---

### 方向 B：思考-验证-反馈 闭环 vs 一次性输出

**天府的做法**：Think（逻辑推演）→ Verify（知识库交叉验证）→ Feedback（用户反馈+RLHF 持续进化）。

**对 AI Judge 的映射**：

```
Think（思考）           Verify（验证）            Feedback（反馈）
分析代码变更          规则库交叉验证              用户确认/驳回
识别争议点           测试用例执行结果             本地微调信号
调用分析器工具        异议Agent挑战               偏好学习
生成初步判断          幻觉检测                    规则权重调整
```

**需要团队回答**：
1. AI Judge 当前是否有「自检/验证」阶段？比如判断生成后，再跑一轮确认规则是否都被正确应用了？
2. 如果用户在 AI Judge 中驳回了某个判断，系统是否有能力记住这个偏好并在未来不再犯同样的「错误」？这个反馈回路是全局的还是按项目隔离的？
3. 验证阶段中，harness 本身就有测试用例——这些 ground truth 是否已经被结构化地用于验证 AI 判断？还是仅作为独立的通过/失败指标？
4. 对于「幻觉检测」——AI Judge 是否有机制确保每个判断都有明确的规则依据，而非模型的自由发挥？

---

### 方向 C：概率化输出（GPRO）vs 确定性判断

**天府的做法**：GPRO 算法对每个推理节点输出置信度，对不确定性诚实表达，而非给出绝对化结论。

**对 AI Judge 的映射**：

```
当前（假设）                    候选方案
"This code has a bug"    →    "Security issue detected
                               Confidence: 95%
                               Similar cases: PR#287 (confirmed)
                               Dissenting view: None"

"Add error handling"     →    "Consider error handling
                               Confidence: 62%
                               Dissenting view: This function is
                               internal-only, error handling may
                               be over-engineering here"
```

**需要团队回答**：
1. AI Judge 当前的评判结果是二元的（通过/不通过）、分数的（7.2/10），还是已经包含了置信度？
2. 如果引入置信度，低置信度的判断应该如何呈现？是降级为「建议」、折叠收起，还是标注为「需人工复核」？
3. 异议 Agent 的引入：是否值得实现一个专门「找茬」的模块？
4. 对于 harness 可以自动验证的维度（比如代码是否通过测试），置信度可以做到 100%；对于主观维度（如代码可读性），置信度天然较低——AI Judge 是否已经有这种维度的区分？

---

### 方向 D：推理链可视化 vs 结果报告

**天府的做法**：从时空校准→排盘→多流派合参→综合决策，每一步在 UI 上展开，用户可点击任意节点查看细节和来源。

**对 AI Judge 的映射**：

```
当前报告（假设）                 候选报告
┌─────────────────┐     ┌─────────────────────────────┐
│ Score: 7.2/10   │     │ ═══ 事实认定 ═══             │
│ Issues: 5       │     │ 变更文件: 5  行数: +147/-38  │
│ - naming (3)    │     │                              │
│ - missing err   │     │ 🔴 安全性 (SAST+AST)         │
│   handling (1)  │     │  payment/checkout.ts:89      │
│ - complexity(1) │     │  SQL注入风险                 │
└─────────────────┘     │  规则: OWASP A03:2021        │
                         │  判例: PR#287                │
                         │  置信度: ████████░ 95%       │
                         │                              │
                         │ 🟡 可维护性 (复杂度分析)      │
                         │  auth/login.ts:45-78         │
                         │  圈复杂度: 18 (阈值: 10)     │
                         │  异议: 业务分支多, 拆分可能   │
                         │        降低可读性            │
                         │  置信度: ██████░░ 62%        │
                         │                              │
                         │ ═══ 合议结论 ═══             │
                         │  阻断: 1  警告: 2  通过: 12  │
                         └─────────────────────────────┘
```

**需要团队回答**：
1. AI Judge 当前的桌面 UI 报告格式是什么？是否已经支持交互式展开/折叠？
2. 推理链可视化在桌面端有天然优势——可以直接点击文件路径在本地 IDE 中打开。当前是否已利用了这个能力？
3. 用户最常质疑 AI Judge 的是哪类判断？这些质疑的本质是否是「我不知道你为什么这么判」？如果是，可视化推理链能否直接解决？
4. 报告中是否需要引入「争议标注」——即明确告诉用户「这个判断存在合理分歧」？

---

### 方向 E：工具矩阵（Function Calling）vs 内置分析器

**天府的做法**：250+ 领域专用工具，覆盖排盘、历法计算、星曜关系、古籍验证等各个环节。

**对 AI Judge 的映射**：

```python
# AI Judge 候选工具矩阵
tools = {
    # 静态分析层
    "sast": SecurityScanner(),        # 安全漏洞
    "ast": ASTAnalyzer(),             # 结构复杂度
    "lint": LintRunner(),             # 代码风格
    "dep": DependencyChecker(),       # 依赖风险

    # 动态验证层
    "test": TestRunner(),             # 测试执行
    "cov": CoverageAnalyzer(),        # 覆盖率
    "perf": PerfBenchmark(),          # 性能基准

    # 知识检索层
    "rules": RuleMatcher(),           # 规则精确匹配
    "precedents": CaseRetriever(),    # 判例向量检索
    "standards": StandardsDB(),       # 行业标准库

    # 对抗验证层
    "challenge": DevilAdvocate(),     # 异议生成
    "hallucination": FactChecker(),   # 幻觉检测

    # 综合层
    "confidence": GPROEstimator(),    # 概率估算
    "reasoning": ReasoningTracer(),   # 推理链记录
}
```

**需要团队回答**：
1. AI Judge 当前的工具/分析器是可插拔架构，还是硬编码的？如果是可插拔的，插件接口的抽象程度如何？
2. 当前的工具覆盖了哪些分析维度？有没有明显的空白（比如安全扫描、性能基准、依赖风险）？
3. 工具调用结果是直接作为「事实」写入判断，还是也经过了模型的二次推理？两种路径各有利弊，当前的设计倾向是什么？
4. 「异议 Agent」本质上也是一个工具——它调用主判断结果作为输入，输出反方论点。这个工具的输入/输出接口应该如何设计？

---

### 方向 F：本地化适配——云端架构 → 本地桌面

**天府的现状**：云端 SaaS，所有推理在服务端完成。

**AI Judge 的场景**：本地桌面运行，数据不出机器。

**需要团队回答**：
1. 如果引入多 Agent 合议 + 思考-验证-反馈闭环，在本地运行的模型规格（参数量、推理速度、内存占用）能否支撑？是需要本地小模型 + 远程大模型的分级策略，还是纯本地？
2. 本地模型微调（用户反馈 → 偏好学习）的技术方案是什么？LoRA？全量微调？还是通过 prompt engineering 实现？
3. 向量数据库（判例检索、规则库）的本地方案：SQLite + 本地 embedding？还是需要更重的方案如 ChromaDB/Qdrant？
4. 桌面端 UI 框架的技术选型：Electron？Tauri？原生？推理链可视化的渲染性能和交互复杂度是否在技术选型的考虑范围内？

---

### 方向 G：harness 特有的优势——天然 Ground Truth

**这是 AI Judge 相对于天府Agent 最大的结构性优势**。

天府Agent 的验证阶段依赖古籍知识库（静态文本），反馈阶段依赖用户主观评价。但 AI Judge 的 harness 天然有：

- 测试用例（通过/失败 = 客观 ground truth）
- Lint 规则（可自动验证）
- 安全扫描结果（可自动验证）
- 覆盖率阈值（可自动量化）
- 性能基准（可自动对比）

**需要团队回答**：
1. 当前 harness 的测试结果等客观数据，是否已经作为「验证信号」反馈给 AI 判断模型？还是仅作为独立报告的一部分？
2. 如果用客观验证信号来做 RLHF（强化学习的 reward signal），这比依赖人类反馈更高效、更可扩展。AI Judge 是否有这方面的尝试或计划？
3. harness 的客观结果是否可以自动校准置信度？比如「测试通过的代码 + AI 说有问题」→ 自动降低该维度判断的置信度？
